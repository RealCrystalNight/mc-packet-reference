0x11 C11PacketEnchantItem

Player selects an enchantment option in the enchanting table window. Carries window ID and button (0=top, 1=middle, 2=bottom).

Notes

The server responds with S2FPacketSetSlot (enchanted item). The server validates the lapis lazuli count and XP level before applying. No major client module specifically targets this packet in the 6+ client codebases examined — enchantment is handled through vanilla GUI mechanics.

implementation Implementation Cases

C11PacketEnchantItem is sent when clicking an enchantment slot in the enchanting table. Button=0 (top, cheapest), 1 (middle), 2 (bottom, most expensive). Server deducts XP + lapis and returns S2FPacketSetSlot with the enchanted item. No hacked client module in Rise 6.0, RektSky B5, Jigsaw 0.26, Sigma 4.11, Memeware 7.3, or Aqua 13 specifically targets C11 — enchantment automation is done through S2D (window open) and S31 (window property) detection followed by PlayerControllerMP window clicks.

AutoEnchant (Vanilla + Client) (Gugustus)

Automatically enchant items by detecting the enchanting table GUI and clicking the best available slot. The module reads enchantment levels from S31PacketWindowProperty.

The module detects S2DPacketOpenWindow for an enchanting table. It reads the three enchantment levels via S31PacketWindowProperty (properties 0, 1, 2). Based on available XP, it picks the best option and clicks via PlayerControllerMP which sends C11 internally.

// Vanilla C11PacketEnchantItem definition: package net.minecraft.network.play.client; import net.minecraft.network.Packet; import net.minecraft.network.PacketBuffer; import net.minecraft.network.play.INetHandlerPlayServer; import java.io.IOException; public class C11PacketEnchantItem implements Packet<INetHandlerPlayServer> { private int windowId; private int button; public C11PacketEnchantItem() {} public C11PacketEnchantItem(int windowIdIn, int buttonIn) { this.windowId = windowIdIn; this.button = buttonIn; } public void readPacketData(PacketBuffer buf) throws IOException { this.windowId = buf.readByte(); this.button = buf.readByte(); } public void writePacketData(PacketBuffer buf) throws IOException { buf.writeByte(this.windowId); buf.writeByte(this.button); } public void processPacket(INetHandlerPlayServer handler) { handler.processEnchantItem(this); } public int getWindowId() { return this.windowId; } public int getButton() { return this.button; } } // AutoEnchant module pattern (Gugustus): // Enchant levels are read from S31PacketWindowProperty: // if (event.getPacket() instanceof S31PacketWindowProperty) { // S31PacketWindowProperty prop = (S31PacketWindowProperty) event.getPacket(); // if (prop.getPropertyId() >= 0 && prop.getPropertyId() <= 2) { // enchantLevels[prop.getPropertyId()] = prop.getValue(); // } // } // // Then click via PlayerControllerMP: // mc.playerController.sendEnchantPacket(windowId, bestButton); // which internally sends: new C11PacketEnchantItem(windowId, bestButton)
Vanilla hook: NetHandlerPlayServer.processEnchantItem(C11PacketEnchantItem) — validates cost, deducts XP + lapis, enchants item, sends S2FPacketSetSlot. Client-side: GuiEnchantment sends C11 when a button is clicked. The levels are shown via S31PacketWindowProperty which the server sends when the enchantment table GUI opens.
No anti-cheat concern — enchanting is PVE. Some servers limit enchant speed to prevent server-load based exploits.
