Notes

Action 0 (ADD_PLAYER): UUID, name, skin properties, gamemode, ping, display name. Action 4 (REMOVE_PLAYER): just UUID. Action 2 (UPDATE_LATENCY): UUID + ping. This is how the client builds the tab list and gets player skins.

implementation Implementation Cases

S38PacketPlayerListItem is the server's mechanism for managing the player tab list. Five actions: ADD_PLAYER (new player joins — sends UUID, name, skin/cape properties, gamemode, ping, display name), UPDATE_GAMEMODE, UPDATE_LATENCY (ping updates every few seconds), UPDATE_DISPLAY_NAME (name changes, nicknames), REMOVE_PLAYER (player leaves). This packet is critical for: AntiBot (verify that a player entity is associated with a real tab list entry), NameProtect (hide certain player names from the tab list), PingSpoof (intercept UPDATE_LATENCY to display fake ping), TabGUI (display player list), and TargetHUD (show target info including ping).

AntiBot / FakePlayer Detector (Rise 6.0, Sigma 4.11, Aqua 13, RektSky B5, Tenacity 5.1)

Verify that player entities in the world have a corresponding tab list entry. If an entity exists but has no S38 ADD_PLAYER entry, it's a bot spawned by the anti-cheat to detect KillAura.

Anti-cheats like Watchdog and Grim spawn invisible player entities (bots) near the player. If KillAura attacks these bots, the anti-cheat knows the player is using an aura. AntiBot maintains a set of valid player UUIDs from S38 ADD_PLAYER actions. When KillAura is looking for targets, it checks if the entity's UUID is in the valid set — if not, the entity is skipped. Additionally, UPDATE_LATENCY is monitored: real players get ping updates, bots don't.

Vanilla hook: NetHandlerPlayClient.handlePlayerListItem(S38PacketPlayerListItem) — processes the action and updates the local player list (tab list). For ADD_PLAYER, it also downloads the player's skin from Mojang's session servers.
Watchdog and Grim both spawn bot entities that DO appear in the tab list (they create fake S38 entries). Better AntiBot modules check for: (1) consistent UPDATE_LATENCY every 2-3 seconds (bots get no ping updates), (2) skin being a default Steve/Alex (bots don't download custom skins), (3) the entity never having custom player metadata. Advanced anti-cheats have started sending fake S38 UPDATE_LATENCY for their bots too.
TabGUI / TargetHUD / Ping Display (Rise 6.0, Sigma 4.11, Aqua 13, Tenacity 5.1)

Read tab list entries to display player names, pings, and gamemodes in the TabGUI or as a KillAura target list. Also used by TargetHUD to show the current target's ping and health.

The module listens for S38 ADD_PLAYER to build its local tab list. For TargetHUD, when KillAura locks onto a target, the module looks up the target's ping from the last UPDATE_LATENCY packet. This is displayed in the TargetHUD overlay (green/yellow/red ping bar).

Vanilla hook: NetHandlerPlayClient.handlePlayerListItem(S38PacketPlayerListItem) — the tab list is a local data structure maintained by this handler.
Some anti-cheats deliberately send fake ping values (e.g., 0 or 9999) for bot entities in UPDATE_LATENCY to confuse ping-based AntiBot detectors. Modern AntiBot modules use multi-factor checks (ping + skin + metadata + movement history) instead of relying on any single signal.
Client-side: NetHandlerPlayClient.handlePlayerListItem(S38PacketPlayerListItem) — this is a complex handler that processes five different action types and maintains the player list data structure. In hacked clients, PacketReceiveEvent fires before this handler. Server-side: Sent by ServerConfigurationManager for player join/leave and by EntityPlayerMP for gamemode changes. UPDATE_LATENCY is sent periodically (every 2-3 seconds) for every online player.
Rise 6.0: AntiBot uses S38 to validate real players. TabGUI reads tab list for display. TargetHUD reads ping from UPDATE_LATENCY. Aqua 13: SessionInfo reads S38 to count online players. Backtrack module may exempt S38 from delay. Sigma 4.11: TabGUI and KillAura both use S38 data for target validation. RektSky B5: No specific module targets S38 — processed through vanilla handler. Tenacity 5.1: KillAura uses S38-based AntiBot for target filtering. Disabler may cancel S38 on certain modes.
