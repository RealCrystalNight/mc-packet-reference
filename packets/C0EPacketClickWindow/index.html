0x0E C0EPacketClickWindow

Player clicks in an inventory window. Handles item moves: pickup, shift-click, number key swap, middle click, drop, drag init, and double-click.

Notes

The server validates every click against its own inventory state — actionNumber must match. If validation fails, S32PacketConfirmTransaction(accepted=false) reverts the client's inventory via S30PacketWindowItems. Almost NO module constructs C0E directly — they call mc.playerController.windowClick() which sends C0E internally. Rise 6.0 BufferAbuseInventoryMove and RektSky B5 ChestStealer are exceptions that construct C0E directly.

implementation Implementation Cases

C0EPacketClickWindow is the most complex serverbound packet — 6 fields controlling ALL inventory item movement. Every inventory module sends C0E, but almost always through the mc.playerController.windowClick() wrapper rather than direct construction. The method calls: windowClick(slot, button, mode) which internally sends new C0EPacketClickWindow(windowId, slotId, button, mode, clickedItem, actionNumber). Each module type (AutoArmor, InventoryManager, AutoPot, ChestStealer) uses different windowClick modes — shift-click (mode=1) for moving items, drop (mode=4) for discarding, number key (mode=2) for hotbar swapping.

BufferAbuseInventoryMove (Rise 6.0) (Rise 6.0)

Abuse C0E buffer desync by sending burst of empty C0EPacketClickWindow() packets to overflow the server's transaction tracking buffer. After overflow, anti-cheat stops validating subsequent inventory clicks.

When clicks % clicksSetting == 0, sends N empty C0E() packets in one pre-motion tick. The server's transaction ID short overflows, desyncing its tracking. All queued legitimate C0E packets are flushed after the burst.

Jigsaw 0.26 also sends a second click to move the old armor to the newly freed slot.

// RektSky B5 - ChestStealer.java (silent C0E construction): short short1 = this.inventory.getNextTransactionID(this.mc.thePlayer.inventory); this.mc.getNetHandler().addToSendQueue( new C0EPacketClickWindow( this.windowId, // stored from S2DOpenWindow index, // slot to steal 0, // left button 1, // mode=1 (shift-click) this.inventory.getSlot(index).getStack(), // ItemStack at slot short1 // transaction ID )); this.inventory.slotClick(index, 0, 1, this.mc.thePlayer); // local sync without sound
Vanilla hook: NetHandlerPlayServer.processClickWindow(C0EPacketClickWindow) — validates transaction ID.
Direct C0E construction avoids PlayerControllerMP's sound/visual updates. Transaction ID must match server's container counter.
Client: PlayerControllerMP.windowClick() is THE entry point. It calls container.getNextTransactionID(), simulates slotClick locally, then sends C0E. Modules rarely construct C0E directly. Server: NetHandlerPlayServer.processClickWindow(C0EPacketClickWindow) re-executes slotClick with provided parameters. If transaction ID doesn't match server's expected next, sends S32PacketConfirmTransaction(accepted=false) and reverts inventory via S30PacketWindowItems. C0E modes used by modules: - mode=1 (shift-click): AutoArmor, ChestStealer, InventoryManager - mode=2 (number key swap): AutoPot, InventoryManager - mode=4 (drop): AutoArmor (drop old armor), InventoryManager (drop junk)
Rise 6.0: BufferAbuseInventoryMove sends empty C0E() burst + queue flush. PingSpoofComponent delays C0E. BadPacketsComponent tracks C0E sends per tick. Sigma 4.11: AutoArmor uses mode=1 shiftClick + mode=4 drop. AutoPot uses mode=2 swap. InventoryManager uses mode=1, mode=2, mode=4. All three import C0EPacketClickWindow. RektSky B5: ChestStealer constructs C0E directly with custom transaction IDs for silent mode. Jigsaw 0.26: AutoArmor uses two windowClick() calls per armor swap. Uses windowClick() for all inventory operations. Aqua 13: Disabler imports C0E for inventory-based bypass modes. Memeware 7.3: No direct C0E construction — uses windowClick().
