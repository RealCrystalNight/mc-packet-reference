0x13 C13PacketPlayerAbilities

Player toggles flying state. Only sent for the flying flag change — the server controls all other ability flags via S39PacketPlayerAbilities.

Notes

The client only sends this when the flying state changes (pressing space twice in creative mode). The server ignores all flags except isFlying for non-creative players. Even if you set allowFlying=true, the server won't let you fly unless it sent S39 with allowFlying=true first.

implementation Implementation Cases

C13PacketPlayerAbilities tells the server the player's ability state — primarily whether they are flying. While it carries invulnerable, flying, allowFlying, creative, flySpeed, and walkSpeed fields, the server typically only respects the isFlying flag for non-creative players. However, this packet is exploited extensively by Disabler modules because sending ability packets with unusual values (NaN flySpeed, extreme walk speed, contradictory flags) can confuse anti-cheat movement validators. Additionally, AbilitiesDisabler sends flying=true to bypass movement checks on servers that auto-disable flight detection when the player claims to be flying.

Disabler — Watchdog / Hypixel (C13 NaN / Extreme Values) (Rise 6.0, Memeware 7.3, Aqua 13, November 0.2)

Send a corrupted C13PacketPlayerAbilities to confuse Watchdog's movement validation, making the anti-cheat unable to determine whether the player should be flying or walking.

The disabler sends C13PacketPlayerAbilities with flySpeed and walkSpeed set to NaN (Float.NaN) or extreme values (Float.MAX_VALUE, -1.0, 0). Watchdog's validator attempts to use these values in speed calculations and produces invalid results, causing it to skip movement checks for the affected player. Some variants send abilities with allowFlying=true even though the server never granted it, creating a state mismatch that certain anti-cheats handle poorly.

Vanilla hook: NetHandlerPlayServer.processPlayerAbilities(C13PacketPlayerAbilities) — the server-side handler. It reads the isFlying flag and applies it IF the player has allowFlying permission. Other flags are only applied for creative-mode players. The EntityPlayerSP sends this when the flying state changes.
Sending NaN or extreme values in flySpeed/walkSpeed can crash older anti-cheat implementations that use these values directly in Java float arithmetic. Modern anti-cheats (Watchdog 2022+, Grim, Intave) validate that speed values are within reasonable ranges (0.0-1.0) before using them. Sending allowFlying=true without server permission is detected by Grim and Intave as 'illegal ability state'.
Disabler — Faithful / Erisium / MMC (Abilities Spam) (Memeware 7.3, Rise 6.0, Aqua 13)

Periodically send abilities packets to keep the anti-cheat's movement validator in a confused state where it can't properly apply gravity and friction checks.

Every N ticks, the disabler sends a C13PacketPlayerAbilities with allowFlying=true and isFlying=true. On servers where the anti-cheat uses the player's flight state to decide movement validation rules, this can cause the anti-cheat to skip movement checks (thinking 'this player is legitimately flying, don't check'). The Disabler mode varies by server: Faithful checks for flySpeed=0.05, MMC spams abilities every 8 ticks, Erisium toggles flying on/off rapidly.

Intave checks that abilities match the server's S39PacketPlayerAbilities response. Some disabler modes only send C13 once on enable and once on disable to reduce detection.
Vanilla Flight / Nuker Flight Toggle (Rise 6.0, Gugustus, Jigsaw 0.26)

Toggle flying legitimately (on servers that grant flight permission) — used by Nuker to reach high blocks and by Flight modules on creative-mode servers.

In vanilla, pressing space twice in creative mode sends C13PacketPlayerAbilities with isFlying=true. Nuker modules use this to fly up to blocks that are out of reach. Some Flight modules also toggle this flag to signal to the server that flight has started/stopped, even if packet-based flight (C04 spoofing) is the actual mechanism.

Vanilla hook: EntityPlayerSP.onUpdateWalkingPlayer() sends C13 when capabilities.isFlying changes. Server receives via NetHandlerPlayServer.processPlayerAbilities().
Toggling flight on servers that don't grant flight permission (no S39PacketPlayerAbilities with allowFlying=true) is detected by all anti-cheats as 'illegal flight'. This is a legitimate feature on creative/free-build servers but is flagged as cheating on survival servers.
Client-side: EntityPlayerSP.onUpdateWalkingPlayer() detects when capabilities.isFlying changes and sends C13PacketPlayerAbilities. On hacked clients, this packet fires through the event system (PacketSendEvent/EventPacket/PacketOutEvent) where modules can intercept it. Server-side: NetHandlerPlayServer.processPlayerAbilities(C13PacketPlayerAbilities) reads the isFlying flag and updates the player's movement state. The server only respects isFlying if allowFlying was previously granted via S39PacketPlayerAbilities. The key insight for Disabler modules: the server uses the player's flySpeed and walkSpeed in movement calculations. By injecting NaN or extreme values, the server's float math produces garbage results, effectively disabling movement-based anti-cheat checks.
