implementation Implementation Cases

C01PacketEncryptionResponse is the client's reply to S01PacketEncryptionRequest during the login handshake. The client generates an AES shared secret, encrypts both the secret AND the server's verify token using the server's RSA public key (from S01), and sends them back. If both values match what the server expects, encryption is enabled for all future packets. This packet is critical for online-mode authentication — without a valid response, the server rejects the connection. Modules target this for: CrackedBypass (cancelling the packet to skip encryption and stay on a cracked server), EncryptionKeyLogger (extracting the shared secret), and Disabler (Aqua 13 disables encryption response in Hypixel mode to bypass anti-cheat verification).

CrackedBypass / AntiEncryption (Jigsaw 0.26, RektSky B5)

Cancel the encryption response packet to prevent encrypted communication. This is useful on servers that have online-mode=true in server.properties but don't actually enforce it, allowing players to join without valid Mojang authentication.

When C01PacketEncryptionResponse is cancelled on the client, the shared secret never reaches the server. The server waits for the encryption response, and if it times out (typically 30 seconds), some outdated server software falls back to unencrypted mode. In practice, most servers will simply disconnect the player. However, on some modified server software and older versions, cancelling this packet can bypass online-mode authentication.

Vanilla hook: NetHandlerLoginClient.handleEncryptionRequest(S01PacketEncryptionRequest) creates the shared secret, authenticates with Mojang, then sends C01PacketEncryptionResponse. Server-side: NetHandlerLoginServer.processEncryptionResponse() decrypts the shared secret, enables AES encryption on the NetworkManager channel.
Cancelling encryption responses is extremely detectable — servers will see a timeout on the encryption handshake. Modern servers use Netty's built-in SSL/TLS and don't rely on the Minecraft encryption protocol. The Aqua 13 Disabler approach (delaying, not cancelling) is more subtle but still easily detected by anti-cheats that monitor packet timing during login.
Backtrack / Disabler Exemption (Aqua 13, Rise 6.0)

Ensures the encryption response is never delayed or buffered by lag-based modules (Backtrack, Blink, PingSpoof), since the server expects it within a specific time window to complete the encryption handshake.

Backtrack buffers incoming/outgoing packets and replays them with artificial delay. C01PacketEncryptionResponse must NOT be delayed because the server's encryption handshake has a tight timeout (typically 5-30 seconds). The module explicitly checks for login/encryption packets and passes them through immediately.

// Aqua 13 - Backtrack.java exempts encryption responses: if (packet instanceof C01PacketEncryptionResponse) { // Pass through immediately - connection critical mc.getNetHandler().getNetworkManager().sendPacket(packet); return; } // Rise 6.0 - Backtrack.java: // Same pattern - exempts all login/handshake packets if (packet instanceof C00Handshake || packet instanceof C00PacketLoginStart || packet instanceof C01PacketEncryptionResponse) { mc.getNetHandler().addToSendQueue(packet); return; }
